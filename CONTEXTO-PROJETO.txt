## MODELO & IMAGEM
- Modelo: deepseek/deepseek-v4-flash (OpenRouter, chave no /opt/data/.env). É barato/fraco em ferramenta —
  se um agente inventar demais mesmo com as travas, considerar subir o modelo.
- Imagem: hermes-agent-sac:latest (base hermes-agent + patches BAKED). Dockerfile em /opt/gastaomatos/hermes/sac-image/.
  RECRIAR o container preserva os patches (estão na imagem); config/env/SOUL/modelo ficam no volume /opt/data.

## TRANSCRIÇÃO DE ÁUDIO (voz do lead -> texto)
- faster-whisper instalado (uv pip na venv; baked na imagem SAC). config.yaml stt.local:
  enabled=true, provider=local, model=SMALL, language='pt'. (model 'base' ALUCINAVA — small+pt acerta.)
- HF_HOME=/opt/data/.cache/hf (volume) -> modelo persiste em recriar, só re-instala o pacote.
- ECHO da transcrição suprimido: o gateway ecoava "🎙️ ..." pro cliente. Patch no whatsapp_cloud.send
  (e bridge.js no Baileys) suprime msg que começa com 🎙️. (Baked na imagem SAC.)

## RAJADA DE MENSAGENS (lead manda várias linhas/áudios -> 1 resposta só, esperando juntar)
- config.yaml display.busy_input_mode = STEER (injeta a 2ª msg na resposta em andamento -> 1 resposta,
  não 2). Opções nativas: interrupt(default)/queue/steer.
- GRACE (espera X seg juntando msgs antes de responder): o grace nativo é só Telegram; estendi pro
  WhatsApp com patch_grace_whatsapp.py + HERMES_..._FOLLOWUP_GRACE_SECONDS=8 no .env.
- config.yaml display.busy_ack_enabled = false (some o "Queued for the next turn").

## TRAVA DE TRAVESSÃO (—)  [regra do dono: proibido em texto público]
- O modelo INSISTE em gerar "—" mesmo com SOUL limpo. Virou TRAVA no código: patch_no_travessao.py
  (baked na SAC) troca —/–/―/‒/− por vírgula ANTES de enviar, em 2 pontos: whatsapp_cloud.send (WhatsApp)
  e send_transac_email (e-mail Brevo). A narração interna ainda tem —, mas NÃO chega ao cliente.

## PORTÕES DE ATENDIMENTO (todo bot de SAC novo)
- WHATSAPP_CLOUD_ALLOW_ALL_USERS=true (atende qualquer número; senão pede "pairing code").
- WHATSAPP_CLOUD_HOME_CHANNEL=<numero do dono> (+_NAME) — cala o prompt "no home channel set".
- config.yaml gateway_restart_notification=false no adapter (não spamma "Gateway shutting down" no dono a cada restart).

## GOOGLE AGENDA
- skill google-workspace (calendar list/create/update/delete). create/update PATCHADOS com sendUpdates="all"
  (dispara o convite pro e-mail do lead). Comando roda com /opt/hermes/.venv/bin/python (tem as libs google).
- config.yaml platform_toolsets: webhook -> [hermes-webhook, terminal]  (SEM o terminal a sessão de E-MAIL
  não roda o google_api.py -> e-mail não agendava; com isso, agenda pelos 2 canais).
- Fuso -03:00 OBRIGATÓRIO no start/end (senão "Missing time zone").
- Auth: OAuth (google_client_secret.json + google_token.json; token chown hermes:hermes senão o gateway não lê)
  OU Service Account (modelo dos NOVOS clientes: cliente compartilha a agenda com o email @...iam.gserviceaccount.com,
  permissão "Fazer alterações nos eventos", igual Analytics). Lib service_account instalada; FALTA adaptar google_api.py.

## E-MAIL BREVO (só casaldotrafego)

- NÃO abrir/acessar links que o lead manda (Nina gastou 87s tentando abrir um Instagram).

- Janela de agendamento: seg-sex, 10h-15h. Preço: ancora no custo de um funcionário CLT, sem inventar número.
- IMPORTANTE: depois de corrigir comportamento, RESETAR a conversa de teste (hermes sessions delete) —

ARQUITETURA MULTI-SERVIDOR — 1 painel só pro dono ver TODOS os clientes (ESTUDO, pra depois)
========================================================================
OBJETIVO: cada cliente roda SOLO no servidor dele (ex.: Hetzner/Alemanha), MAS o dono entra em UM painel
só (agente-ia.casaldotrafego.com) e vê/controla TODOS. Sem "1 link por cliente" (o dono rejeitou isso).

RECOMENDADO = BANCO CENTRAL + SYNC de cada servidor (estende o que JÁ existe, não reinventa):
- Hoje os 3 agentes deste VPS já sincronizam pro Neon central (schema por cliente) e o dashboard lê o Neon.
  O modelo multi-servidor é O MESMO: o sync roda em CADA servidor remoto apontando pro MESMO Neon central.
- Cada servidor de cliente roda: (a) o bot (container Hermes), (b) o SYNC (central-db/sync.py) empurrando
  conversas/leads pro Neon central no schema DAQUELE cliente, (c) uma CONTROL API (tipo agente_painel) no
  domínio do cliente, pra pausar/responder.
- O dashboard central ganha um REGISTRO de agentes: {slug, schema, url_da_control_api, page_id, ...}.
  Cadastra o cliente novo no registro -> ele aparece junto dos outros. UM painel, todos os clientes.
- Ações de controle (pausar/enviar) roteiam pra control API DAQUELE servidor (cada um no seu domínio).
  O dashboard guarda a URL da control API por agente. Leitura vem toda do Neon central.
- Isolamento: schema por cliente no Neon central (regra que já vale). Nunca mistura.

ALTERNATIVAS DESCARTADAS:
- 1 dashboard por cliente (subdomínio cliente.agente...): dono rejeitou (vira 1 milhão de links).
- Dashboard federado (consulta a API de cada servidor ao vivo): mais complexo/lento; central+sync é melhor.

O QUE MUDA NO MOLDE DE CLIENTE-SOLO (Parte B): além do bot, subir o SYNC (apontando pro Neon central,
schema do cliente) + a CONTROL API (no domínio do cliente) + registrar o cliente no dashboard central.
Cada servidor remoto precisa: URL do Neon central (segredo) + token da própria control API.
NÃO implementar agora — é o caminho de escala. A base (Neon central + sync + control API) já está pronta;
só falta DISTRIBUIR o sync/control por servidor e o dashboard virar multi-tenant por registro.
